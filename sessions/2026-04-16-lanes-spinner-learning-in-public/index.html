Lanes, Spinner, and learning in public

Ben Miller

April 16, 2026

So I just want to try an experiment having to do with how I develop. For a while I had a process of starting each session where I’d just sit down and type out the current context of the project and what I intend to work on. I sort of stopped doing that but I’d like to get back into it again, only this time using dictation software (WisprFlow, which I’m currently using).

So let’s see — what are the things I’m most frustrated by right now?

  1. I think the Sheetwork project has gotten way out of hand.
  2. I tried doing an overnight agent run. I gave it a big, ambitious set of tasks and let it run for a couple of hours. I don’t know how long it even took — it might have been under 20 minutes — but it created a kind of drift that I’m still partially recovering from. Basically I produced all this code that I had to review and clean up, and because it drifted so far from the main branch, it almost took more time than if I’d been at the wheel the entire time. With each small change I could have made sure it didn’t drift too far from my intentions.

I’m just trying to lay out what’s on my mind about this project: the things that are bothering me, the things that are slowing me down, the things that are distracting me. It feels like brain mush whenever I sit down at my computer because there are so many moving parts, so many things going on, so many projects I start and don’t finish.

The context-switching problem #

The main obstacle is what to do in the time when a given prompt is running. I don’t want to sit there watching it, of course — it’s better to context switch and focus on something else. That’s maybe the problem I’m trying to solve: context switching between projects, reading in between, whatever. Even then, reading in between is weird. There’s a kind of shallow focus I can only access when I’m doing something like this. If I actually need to read something or do something requiring deep focus, it would have to be when I’m not vibe coding.

The Lanes project #

So anyway, what I’ve been working on in parallel is a project called Lanes — named after the lanes on a highway. The idea is switching between lanes: basically a context switcher or workspace switcher, applied to the broader scope of applications, windows within applications, panes, tabs, whatever I can access and control. Just having a way of defining these things, saving them, and switching between them.

It’s nothing fundamentally new, really, but the main differentiating factors are that it’s backed by an Elixir server, so it could go beyond just what’s on the desktop — maybe controlling what you have on your phone, in the browser, in web apps, stuff like that.

It definitely has the capacity to grow in scope indefinitely. It’s also fairly easy to vibe code, so it’s kind of a huge distraction risk. I’ll be honest about that.

The Spinner utility #

A smaller component of this — and actually the original thing I was working on that started to turn into Lanes — is a utility I’m calling Spinner. It basically lets me spin up web servers per worktree. For example, in the Sheetwork application: create a worktree, CD into it, run spinner up. It prompts me to run spinner setup, which:

  • Creates a test database
  • Creates a local database configuration
  • Seeds the data

It makes it so I can run the application from that directory within a namespace specific to that worktree.

The Spinner script itself is fairly narrow in scope. I’m honestly surprised nothing like it exists — though I did run it past the AI multiple times before building it to confirm I wasn’t reinventing the wheel. From what I can tell, nothing quite like what I need exists. And because it’s small in scope, I’m not worried about it blowing up.

The bell hook #

One killer feature I wanted for Lanes right now was the ability to hook into Claude. The way I have it configured is a hook that runs a bell sound whenever the prompt returns — super useful because I don’t have to sit there checking every session visually to see if it’s returned. I can just listen for it, then return to it. It’s really nice because I can focus on other things while the prompt runs.

I was thinking it might be possible to not just hear the bell, but automatically have it take you to that context. That could get annoying though. A better idea: instead of forcing the context switch, add it to a queue. When I’m ready to look at another Claude window, I hit a shortcut and I’m at the next one demanding attention. That would be really nice — but the more I think about it, the more I realize it needs to be its own thing.

The plan #

As tempting as it is, I should defer digging into Lanes until Sheetwork is in a better state. The Spinner utility is worth finishing since it’s mostly there already. For Lanes proper — time to put it on the backlog.

So:

  1. Clean up Spinner and keep it separate from the broader Lanes project.
  2. Abandon the Lanes project for now (backlog it).
  3. Resume focus on Sheetwork.

Learning in public #

One major thing I’ve been neglecting: learning in public. If the purpose of this project were to deploy it and turn it into a product, that would merit one approach — but that’s not what I’m intending. I want to use it as a portfolio piece and a learning project. It might not even be that impressive as a portfolio piece, but if I can document the journey and develop it somewhat in public — blog posts per release, that sort of thing — I think that’s where the main value comes from.

I’m not doing any of that, and I need to fix it. Now that I have access to WisprFlow, it opens up a lot of opportunities for thinking in public. I can get my thoughts out much more quickly than I ever have before. This is the time to get a personal blog set up and start posting — not necessarily every day, but once per release, whatever cadence makes sense.

I also need to start thinking about releasing Sheetwork. As part of that, I need at least a basic release process template. Even if there’s only one release, I want some meta-process for releasing things into the public in a way that can be demonstrated to people and has value as a way of promoting my work or demonstrating my thought process.

Summary #

  • Ditch Lanes (backlog it).
  • Keep Spinner — minimal scope, mostly done.
  • Learn in public more, especially now that dictation makes it easy.
  • Clean up Sheetwork, get it to a releasable state, document the process.
  • Get the personal website up and running.

None of this is actually that hard at this point. It’s all really straightforward.
