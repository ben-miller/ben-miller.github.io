Sheetwork UI planning

Ben Miller

March 6, 2026

Plan mode. Wanting to implement some major UI features, and to do this I want to make sure we start on the right footing. Currently the main UI is at http://localhost:4000/admin/inspect/2026-03-02, implemented using Phoenix LiveView. It’s an “inspection” endpoint, which suggests we’re probably abusing the term and URL.

We’re pretty early on in the project, so I’m open to a complete restructuring — even abandoning Elixir/Phoenix if it makes sense to. I’m mostly happy with how the DB is, and so it should be easy to write a new application that wraps itself around our current schema.

What I envision for the UI:

  • Quarterly view (by quarter)
    • Quarterly overview
    • Quarterly planner
  • Weekly view (by week)
    • Weekly overview
    • Weekly planner
    • Weekly audit/reflection

Maybe also a daily view.

This would essentially involve tabs. Not sure how to structure this because there are potentially many weeks, quarters, days. Does the daily/weekly/quarterly view focus on just the current period? What happens if we want to go back in time or into the future? Don’t want to think too hard about this yet — but at least I want to get the model right.

For now, maybe just a flat list of tabs for the five views listed above. Within each view it should be possible to time-travel to past or future periods, with today/this week/this quarter as the default.

Currently focusing on the weekly view. I envision:

  • Weekly overview
    • Calendar (daily schedule)
    • The info we’re currently displaying for “important”, “week slots”, and “week suggestions”
    • No PDF on the right
  • Weekly planner
    • PDF on the right, as it is now in the inspection view
    • The inbox and un-tagged containers we currently have in the inspection view
    • A “next week” section for dragging items into next week’s inbox

Yesterday we made a model for “container” — essentially a Trello-like space a task (card) can be dragged into. Containers for the weekly view might include:

  • Inbox
  • For each day of the week (Mon–Sun), a set of available time slots:
    • morning_tasks
    • shibuya_am
    • shibuya_pm
    • kichijoji_am
    • at_school
    • kichijoji_pm
    • in_eifukucho
    • combini_pm
    • evening_tasks
    • outings
    • night_out
    • cook_dinner
  • Next week (inbox of next week)

These slots are registered in weekly.yml. We need logic to present the availability of each slot — for example, night_out makes sense on a Saturday, and shibuya_am makes sense on weekdays or when a day is tagged as class.

In general, I want to use tags as a way for slots to advertise what they have to offer, and for tasks to define what they require in a time slot. For example, a dev session task might need 3+ hours of deep concentration and no distractions — tagged deep-work. Or a task like “thrift store scan” might take 10 minutes and require being in a certain location — tagged requires:in_shibuya, takes:10m. Slots might have corresponding tags like offers:in_shibuya or offers:3hrs. Getting into predicate logic territory — not sure if that’s a good thing yet.

This tag-matching system — used to generate could-do lists — is out of scope for now, but worth keeping in mind as we design the model.

Goals for this session:

  • Discuss architecture, consider alternatives to the current stack.
  • Separate current functionality into weekly planning and weekly overview tabs.
  • Define logic for filtering which slots are available on which day. Should be straightforward — probably some new YAML fields. (Not the full slot-matching concept described above.)
