How I structure my notes

2025/07/29

Like many, I suffer from the habit of digital hoarding. Whenever I come across an idea, future plan, or piece of useful information, I add it to my ever-growing system of notes, now exceeding 10,000. As this continues to grow, every few years I need to re-think how to structure my collection of notes to so that it doesn’t collapse under the weight of itself.

I’ve gone through numerous phases with this. I’ve explored and/or migrated between various notetaking tools, including Evernote, Org Mode, Roam Research, Logseq, Notion and, currently, Obsidian. I’ve read a book on the Zettelkasten method, and have adopted a loose interpretation of the PARA system. Like many digital hoarders, tending to this “digital garden” has been an ongoing hobby of mine.

One huge problem I’ve faced as my notes have grown is how to structure the directories. There are numerous ways you could go about this. You could have a single directory containing thousands of notes, you could alphabetize everything, you organize based on dates, you could divide everything into topics using some Dewey Decimal-esque classification system. But what I’ve found to work better than anything else is to use a directory structured based on Basic Formal Ontology (BFO).

The basic idea is that every note I add to the collection should have a clearly defined place to go. When adding a new note, I start by asking myself what the note actually represents: is it for a website, person, concept, physical location, cultural event, technique, tool, etc.? Based on that, I should be able to know pretty quickly where exactly in the directory structure to put the note - or whether a new category needs to be added, and where exactly to put that category. I do this in a topic-agnostic way, letting hashtags deal with topic labelling.

This can be done by having a directory tree of is-a relationships that each entity can fit nicely and unambiguously into. Each note should have an is-relationship with its directory, and each directory has an is-a relationship with its parent, all the way up to the root entity. Defining what these is-a relationships up to the root level is actually pretty hard to do, but BFO provides a kind of scaffold for this which has worked remarkably well for the purpose of organizing notes logically.

In my case, I made a repo called “Ontology” which I use as a kind of personal wiki, or conceptual map of the world. I use BFO to define the root directories, and I’ve created my own ontology by extending that with a system of subdirectories. For each note that I create, it’s usually pretty obvious where to put it. When it’s not obvious, I add a new subdirectory, often employing ChatGPT to come up with a good directory path for it.

It’s an eternal work-in-progress, and far from “BFO-compliant”, but it has nonetheless worked extremely well for keeping up with the mass of notes that I’ve continued to accumulate over the years. Just throwing that out there as a possible way of taming your digital garden.
